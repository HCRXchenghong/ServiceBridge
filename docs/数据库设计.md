# 数据库设计

当前后端提供内存仓库用于本地演示和单元测试，商业部署使用 PostgreSQL + Redis。`backend/migrations/001_init.sql` 是第一版关系模型基线，PostgreSQL 仓库启动时也会自动补齐已新增字段和表。

## 核心原则

- `visitors.ip` 和 `conversations.visitor_ip` 保存原始 IP，不被备注覆盖。
- `visitor_remark` 只作为运营备注展示，可由负责客服或管理员修改。
- 消息以 `messages.server_msg_id` 为服务端唯一 ID；客户端重试时用 `client_msg_id` 做幂等去重。
- 会话同一时间只允许一个 `assigned_agent_id`，转接必须先释放旧客服计数，再写入新客服。
- AI Key 不明文回传。配置 `DATA_ENCRYPTION_KEY` 后数据库字段会加密存储，接口只返回脱敏值。

## PostgreSQL 表

- `admin_users`：管理账号。
- `agents`：客服账号、分组、在线状态、接待上限。
- `visitors`：访客身份、原始 IP、默认备注。
- `conversations`：会话状态、分配客服、未读计数、访客备注。
- `messages`：文本、表情、联系方式、AI、系统、转人工消息。
- `push_devices`：客服 App 推送设备 token。
- `keyword_rules`：关键词自动回复规则。
- `contact_settings`：电话、微信、QQ 等对外展示配置。
- `ai_settings`：OpenAI 原生或兼容接口配置。
- `business_hours`：人工营业时间。
- `auth_sessions`：登录 token 和访客 token。
- `audit_events`：管理与客服关键操作审计。

## Redis 用途

- `ws:conn:{node_id}:{client_id}`：当前节点连接注册，设置短 TTL。
- `agent:online:{agent_id}`：客服在线/忙碌状态。
- `agent:load:{agent_id}`：客服当前接待计数。
- `conversation:lock:{conversation_id}`：派单、转接、关闭会话的短锁。
- `pubsub:events`：跨节点广播实时事件。

## 5000 并发部署建议

- 后端实例无状态横向扩容，Nginx 对 `/ws` 使用长连接反代。
- 单实例先按 2500 WebSocket 连接做容量估算，两实例起步；生产建议至少三实例，滚动发布时保留冗余。
- WebSocket 心跳间隔 25-30 秒，服务端读超时大于两倍心跳。
- Postgres 承担强一致数据和派单事务；Redis 承担跨节点实时事件广播。
- 消息写入先落库，再广播；广播失败不影响历史记录，客户端重连后用历史接口补齐。
