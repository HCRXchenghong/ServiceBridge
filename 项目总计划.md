# 项目总计划

本文档是客服系统的执行计划。边界以 `客服系统边界说明.md` 为准，本文档负责拆阶段、拆模块、定开工顺序。

## 1. 总目标

做一个轻量、高效、纯自研的在线客服系统。

三端形态：

- 用户端：Web 页面，支持电脑浏览器和手机浏览器。
- 客服端：uni-app，支持 Android 和 iOS App。
- 管理端：uni-app，支持 Android 和 iOS App。

UI 原则：

- 用户端按 `用户端前端UI示例.html` 一比一实现。
- 客服端按 `客服端前端UI示例.html` 一比一实现。
- 管理端按 `管理端前端UI示例.html` 一比一实现。
- 电脑浏览器上的用户端可以在现有手机外壳基础上优化尺寸、居中和窗口适配，但视觉风格不能变。

性能目标：

- 不考虑带宽瓶颈时，系统按 5000 并发 WebSocket 连接容灾设计。
- WebSocket 节点可横向扩容。
- 消息先落库再投递。
- 单节点异常时不丢已确认消息。

## 3. 后端模块

后端使用 Go。

核心模块：

- 账号模块：管理账号、客服账号、登录、鉴权、权限。
- 访客模块：访客 ID、访客 IP、默认备注、备注修改。
- 会话模块：会话创建、状态流转、当前负责人唯一。
- 派单模块：在线客服、接待上限、最少会话优先、轮询。
- 消息模块：文本、表情、联系方式、系统消息、AI 消息。
- WebSocket 网关：鉴权、心跳、重连、消息投递。
- AI 模块：OpenAI 原生 API 和 OpenAI 兼容协议适配。
- 关键词模块：关键词自动回复、人工客服保留关键词。
- 营业时间模块：人工优先、非营业时间 AI、超时转 AI。
- 通知模块：客服 App 前台提醒、提示音、后台推送抽象。
- 管理模块：配置、会话监控、客服管理、聊天记录。

## 4. 前端模块

### 4.1 用户端 Web

第一屏就是客服窗口，不做营销落地页。

必须实现：

- 手机浏览器全屏聊天体验。
- 电脑浏览器居中手机外壳体验。
- 文本、表情、更多面板。
- 电话、微信、QQ 联系方式触发。
- AI / 人工 / 排队 / 已结束状态。
- 转人工。
- 提示音开关。
- 历史消息加载入口。
- 服务评价。

### 4.2 客服端 uni-app

必须实现：

- 登录。
- 会话列表。
- 上线、忙碌、离线切换。
- 聊天详情。
- AI 接管状态下的人工接管按钮。
- 访客 IP 和备注展示。
- 修改备注。
- 快捷回复。
- 发送电话和微信。
- 结束会话。
- 前台横幅通知。
- 提示音和震动配置。
- 推送 Token 注册和后台系统通知 webhook 接入。

### 4.3 管理端 uni-app

必须实现：

- 管理账号登录。
- 数据概览。
- 会话监控。
- 会话详情只读查看。
- 强制转接。
- 强制结束。
- 客服账号管理。
- 客服分组和接待上限。
- AI 配置。
- OpenAI 兼容接口配置。
- AI 提示词配置。
- 系统联系方式配置。
- 微信 / QQ 图文或纯文本回复配置。
- 营业时间配置。
- 超时阈值配置。
- 关键词自动回复入口。

## 5. 开发阶段

### 阶段一：项目骨架

目标：

- 建立后端、用户端、客服端、管理端目录。
- 保存和归档三个 UI 示例。
- 确定配置文件、环境变量和本地运行方式。

完成标准：

- 目录结构稳定。
- 三端都有独立入口。
- 后端能启动健康检查接口。

### 阶段二：基础账号和会话

目标：

- 管理账号登录。
- 客服账号登录。
- 访客创建会话。
- 记录访客 IP。
- 默认备注为访客 IP。
- 客服可修改备注。

完成标准：

- 管理端、客服端能登录。
- 用户端打开后能创建会话。
- 会话详情能看到 IP 和备注。

### 阶段三：WebSocket 消息链路

目标：

- 访客 WebSocket。
- 客服 WebSocket。
- 消息发送、落库、投递。
- 心跳。
- 断线重连补拉。

完成标准：

- 用户端和客服端可以互发文字和表情。
- 消息有 `client_msg_id` 和 `server_msg_id`。
- 断线重连后历史消息不丢。

### 阶段四：多客服派单

目标：

- 客服在线、忙碌、离线。
- 接待上限。
- 最少会话优先派单。
- 会话当前负责人唯一。
- 客服只能看到自己负责会话。

完成标准：

- 多客服在线时，新会话只派给一个客服。
- 其他客服不可见未分配会话。
- 管理端可查看所有会话。

### 阶段五：AI、营业时间和关键词

目标：

- 关键词自动回复。
- 人工客服保留关键词。
- OpenAI 适配层。
- AI 提示词。
- 营业时间。
- `human_first`、`always_ai`、`manual_only` 模式。
- 客服超时未回复自动转 AI。

完成标准：

- 营业时间内默认人工优先。
- 非营业时间可 AI 接待。
- 一直 AI 模式可用。
- 用户输入“人工客服”等关键词会提醒客服。
- 超过配置秒数未回复可自动转 AI。

### 阶段六：App 通知

目标：

- 客服端前台横幅。
- 提示音。
- 震动。
- 通知权限引导。
- 后台系统通知抽象。
- 推送 Token 注册。

完成标准：

- App 打开时有弹窗和提示音。
- App 后台时有系统通知能力。
- 点击通知可进入对应会话。

### 阶段七：5000 并发和容灾

目标：

- 多 WebSocket 节点。
- Redis 在线状态和连接映射。
- 跨节点投递。
- 派单锁和原子计数。
- 压测脚本。
- Nginx WebSocket 反代配置。

完成标准：

- 5000 并发连接压测不崩溃。
- 心跳稳定。
- 消息投递稳定。
- 单节点异常后可重连恢复。

## 6. 第一轮开工顺序

第一轮从低风险骨架开始：

1. 建立项目目录。
2. 建立 Go 后端基础服务。
3. 建立用户端 Web 入口。
4. 建立客服端 uni-app 入口。
5. 建立管理端 uni-app 入口。
6. 把三个 HTML 示例归档为 UI 基准。
7. 写本地启动说明。

## 7. 不变原则

- UI 不能随意改风格。
- 消息必须先落库再投递。
- 会话当前负责人唯一。
- 人工客服关键词优先级最高。
- 营业时间内默认人工优先。
- 客服不可见未分配给自己的会话。
- 管理账号权限最大。
- 5000 并发按横向扩容和容灾设计，不按单机硬扛设计。
